# Biddy 코딩 컨벤션

## 네이밍 규칙

| 대상 | 규칙 | 예시 |
|---|---|---|
| 패키지 | 소문자 | `com.biddy.member` |
| 클래스 | PascalCase | `MemberService` |
| 메서드/변수 | camelCase | `findById()` |
| 상수 | UPPER_SNAKE | `MAX_BID_AMOUNT` |
| DB 테이블 | snake_case | `member_info` |
| DB 컬럼 | snake_case | `created_at` |
| REST URI | kebab-case, 복수형 | `/api/products` |

## 패키지 구조 (도메인 서비스 공통)

```
com.biddy.{service}/
├── controller/     ← REST 컨트롤러
├── service/        ← 비즈니스 로직
├── repository/     ← JPA Repository
├── entity/         ← JPA 엔티티
├── dto/            ← 요청/응답 DTO
│   ├── request/
│   └── response/
├── config/         ← 설정 클래스
├── exception/      ← 커스텀 예외
└── kafka/          ← Kafka Producer/Consumer
```

## Git 커밋 메시지

```
<type>: <subject>

type:
  feat     - 새로운 기능
  fix      - 버그 수정
  refactor - 리팩토링
  docs     - 문서 수정
  test     - 테스트 추가/수정
  chore    - 빌드, 설정 변경
```

## 브랜치 전략

| 브랜치 | 용도 |
|---|---|
| `main` | 배포 가능한 안정 버전 |
| `develop` | 개발 통합 브랜치 |
| `feature/{서비스명}/{기능}` | 기능 개발 |
| `fix/{서비스명}/{설명}` | 버그 수정 |

# 도메인 간 이벤트 흐름 정의

> 모든 필드명은 실제 테이블 컬럼명 기준으로 작성.
> `auction` 테이블, `bid` 테이블 필드를 직접 참조한다.

---

## 1. Auction Service 이벤트

### 1.1 입찰 발생 (BID_PLACED)

| 흐름 | 현재 방식 | 대안 | 대상 | 필요한 데이터 (테이블.컬럼) | 이유 |
|------|----------|------|------|--------------------------|------|
| Auction → WebSocket | WebSocket push | **Redis Pub/Sub + WebSocket** (다중 인스턴스 시) | 경매 구독자 전체 (현재가 갱신) | `auction.auction_id`, `auction.current_bid`, `auction.bid_count`, `bid.bidder_id` | 실시간 현재가 갱신. 단일 서버면 현재 방식 OK, 스케일아웃 시 Redis Pub/Sub으로 모든 WS 서버에 전파 필요 |
| Auction → WebSocket | WebSocket push | 동일 | 이전 최고 입찰자 (경쟁 패배 알림) | `auction.auction_id`, `bid.bidder_id` (이전), `bid.amount` | 경매 페이지 구독 중이면 WebSocket으로 직접 push. **Member Service 경유 불필요** |
| Auction → WebSocket | WebSocket push | 동일 | 입찰자 본인 (입찰 확인) | `auction.auction_id`, `bid.bidder_id`, `bid.amount` | 경매 페이지 구독 중이면 WebSocket으로 직접 push. **Member Service 경유 불필요** |

> **MVP 단계:** 경매 페이지를 보고 있는 사용자는 WebSocket으로 직접 알림. 다른 도메인 경유 불필요.
> **성장 단계:** 페이지를 떠난 사용자에게도 알림이 필요하면 Notification Service 추가 (Kafka → FCM/APNS 푸시).

**이벤트 Payload (bid + auction 테이블 조합)**

```json
{
  "eventType": "BID_PLACED",
  "timestamp": "2026-06-12T13:55:00",
  "payload": {
    "auctionId": "A-FNF97",        // auction.auction_id
    "bidId": 101,                   // bid.bid_id
    "bidderId": 42,                 // bid.bidder_id
    "amount": 740000,               // bid.amount
    "bidAt": "2026-06-12T13:55:00", // bid.bid_at
    "currentBid": 740000,           // auction.current_bid (갱신 후)
    "bidCount": 7,                  // auction.bid_count (갱신 후)
    "previousBidderId": 35          // 직전 최고 입찰 bid.bidder_id (없으면 null)
  }
}
```

| payload 필드 | 출처 테이블.컬럼 | 설명 |
|-------------|----------------|------|
| `auctionId` | `auction.auction_id` | 경매 식별자 |
| `bidId` | `bid.bid_id` | 이번 입찰 PK |
| `bidderId` | `bid.bidder_id` | 입찰자 회원 ID |
| `amount` | `bid.amount` | 입찰 금액 |
| `bidAt` | `bid.bid_at` | 입찰 시각 |
| `currentBid` | `auction.current_bid` | 갱신된 현재 최고가 |
| `bidCount` | `auction.bid_count` | 갱신된 총 입찰 수 |
| `previousBidderId` | 직전 최고 `bid.bidder_id` | 경쟁 패배 알림 대상 (첫 입찰이면 null) |

### 1.2 경매 종료 — 낙찰 (AUCTION_ENDED / SOLD)

| 흐름 | 현재 방식 | 대안 | 대상 | 필요한 데이터 (테이블.컬럼) | 이유 |
|------|----------|------|------|--------------------------|------|
| Auction → Order | Kafka (비동기) | **Kafka + Outbox Pattern** | Order Service | `auction.auction_id`, `auction.seller_id`, `auction.current_bid`, `auction.name`, `auction.brand`, `auction.thumbnail_url`, 낙찰자 `bid.bidder_id` | 낙찰 주문 자동 생성. **Outbox로 이벤트 유실 방지** (돈 관련 흐름이므로 필수) |
| Auction → Payment | Kafka (비동기) | **REST (동기) + Saga 보상** | Payment Service | `auction.auction_id`, 낙찰자 `bid.bidder_id`, `auction.current_bid` | 결제 준비는 반드시 성공해야 함. REST로 즉시 실패 감지 → 보상 트랜잭션. 현재 규모에서는 Kafka + Outbox로 충분 |
| Auction → WebSocket | WebSocket push | **Redis Pub/Sub + WebSocket** (다중 인스턴스 시) | 경매 구독자 전체 | `auction.auction_id`, 낙찰자 `bid.bidder_id`, `auction.current_bid` | 실시간 종료 알림 + 낙찰자/판매자 알림. 경매 페이지 구독 중이면 **WebSocket으로 직접 push** |

> **MVP 단계:** 낙찰자/판매자가 경매 페이지를 보고 있으면 WebSocket ENDED 메시지로 직접 알림.
> **성장 단계:** 페이지를 떠난 낙찰자에게 "24시간 내 결제" 알림이 필요하면 Notification Service 추가 (Kafka → FCM/이메일).

**이벤트 Payload (auction + bid 테이블 조합)**

```json
{
  "eventType": "AUCTION_ENDED",
  "timestamp": "2026-06-12T15:30:00",
  "payload": {
    "auctionId": "A-FNF97",           // auction.auction_id
    "status": "SOLD",
    "sellerId": 10,                    // auction.seller_id
    "winnerId": 42,                    // 최고 입찰 bid.bidder_id
    "finalBid": 740000,                // auction.current_bid (종료 시점)
    "productName": "나이키 덩크 로우",  // auction.name
    "brand": "Nike",                   // auction.brand
    "edition": "Low Retro",            // auction.edition
    "category": "sneakers",            // auction.category
    "thumbnailUrl": "https://...",     // auction.thumbnail_url
    "startPrice": 400000,              // auction.start_price
    "totalBids": 7,                    // auction.bid_count
    "endsAt": "2026-06-12T15:30:00",   // auction.ends_at
    "paymentDeadline": "2026-06-13T15:30:00"  // ends_at + 24h (계산값)
  }
}
```

| payload 필드 | 출처 테이블.컬럼 | 소비자 | 용도 |
|-------------|----------------|--------|------|
| `auctionId` | `auction.auction_id` | Order, Payment, Member | 연결 키 |
| `sellerId` | `auction.seller_id` | Order, Member | 판매자 식별, 알림 |
| `winnerId` | 최고 입찰 `bid.bidder_id` | Order, Payment, Member | 구매자 식별 |
| `finalBid` | `auction.current_bid` | Order, Payment | 결제 금액 |
| `productName` | `auction.name` | Order | 주문 내역 상품명 |
| `brand` | `auction.brand` | Order | 주문 내역 브랜드 |
| `edition` | `auction.edition` | Order | 주문 내역 에디션 |
| `category` | `auction.category` | Order | 주문 분류 |
| `thumbnailUrl` | `auction.thumbnail_url` | Order | 주문 내역 이미지 |
| `startPrice` | `auction.start_price` | Order | 주문 내역 참고 |
| `totalBids` | `auction.bid_count` | Order | 주문 내역 참고 |
| `endsAt` | `auction.ends_at` | Order | 경매 종료 시각 기록 |
| `paymentDeadline` | `ends_at + 24h` | Payment | 결제 기한 타이머 |

### 1.3 경매 종료 — 유찰 (AUCTION_UNSOLD)

| 흐름 | 현재 방식 | 대안 | 대상 | 필요한 데이터 (테이블.컬럼) | 이유 |
|------|----------|------|------|--------------------------|------|
| Auction → WebSocket | WebSocket push | **Redis Pub/Sub + WebSocket** (다중 인스턴스 시) | 경매 구독자 전체 + 판매자 | `auction.auction_id`, `auction.seller_id`, reason, winnerId: null | 실시간 종료 알림. 판매자가 경매 페이지를 보고 있으면 **WebSocket으로 직접 유찰 알림**. Member Service 경유 불필요 |

> **MVP 단계:** WebSocket ENDED 메시지로 판매자에게 직접 유찰 알림.
> **성장 단계:** 판매자가 오프라인이면 Notification Service로 "경매가 유찰되었습니다" 푸시/이메일.

**이벤트 Payload (auction 테이블)**

```json
{
  "eventType": "AUCTION_UNSOLD",
  "timestamp": "2026-06-12T15:30:00",
  "payload": {
    "auctionId": "A-FNF97",        // auction.auction_id
    "sellerId": 10,                 // auction.seller_id
    "productName": "나이키 덩크 로우", // auction.name
    "endsAt": "2026-06-12T15:30:00", // auction.ends_at
    "reason": "NO_BID"              // NO_BID | PAYMENT_TIMEOUT
  }
}
```

| payload 필드 | 출처 테이블.컬럼 | 용도 |
|-------------|----------------|------|
| `auctionId` | `auction.auction_id` | 경매 식별 |
| `sellerId` | `auction.seller_id` | 판매자 알림 대상 |
| `productName` | `auction.name` | 알림 메시지 내 상품명 |
| `endsAt` | `auction.ends_at` | 종료 시각 기록 |
| `reason` | 비즈니스 로직 판단 | `NO_BID` (입찰 없음) / `PAYMENT_TIMEOUT` (미결제) |

## 2. Payment Service 이벤트

### 2.1 결제 완료 (PAYMENT_COMPLETED)

| 흐름 | 현재 방식 | 대안 | 대상 | 필요한 데이터 | 이유 |
|------|----------|------|------|-------------|------|
| Payment → Order | Kafka (비동기) | **Kafka + Outbox Pattern** | Order Service | orderId, `auction.auction_id`, paidAmount, paidAt | 결제 완료 → 주문 상태 PAID. Outbox로 원자성 보장 |
| Order → Auction | Kafka (비동기) | **Kafka + Outbox Pattern** | Auction Service | `auction.auction_id`, orderId, `bid.bidder_id` (winnerId), paidAmount | **Order가 중계하여 Auction에 거래 완료 전달**. Payment가 Auction을 직접 호출하지 않음 |
| Payment → Settlement | 없음 (추가 필요) | **Kafka + Outbox Pattern** | Settlement Service | orderId, paidAmount, sellerId | 정산 데이터 생성. 결제 완료와 정산이 원자적이어야 함 |
| Payment → 프론트 | 없음 (추가 필요) | **WebSocket** 또는 **Notification Service** | 낙찰자/판매자 | `bid.bidder_id`, `auction.seller_id`, paidAmount | 결제 완료 페이지에 있으면 WebSocket 직접 push. 오프라인이면 Notification Service (FCM/이메일) |

> **결제 완료 흐름:** Payment → Order → Auction (체인 방식)
> Payment는 Order에만 결제 결과를 전달하고, Order가 주문 상태 갱신 후 Auction에 거래 완료를 전파한다.
> Payment가 Auction을 직접 알 필요가 없으므로 **서비스 간 결합도가 낮아진다.**

```
Payment Service                Order Service               Auction Service
     │                              │                            │
     │  PAYMENT_COMPLETED           │                            │
     │  (orderId, paidAmount)       │                            │
     ├─────────────────────────────>│                            │
     │                              │  order.status = PAID       │
     │                              │                            │
     │                              │  ORDER_PAID                │
     │                              │  (auctionId, orderId,      │
     │                              │   winnerId, paidAmount)    │
     │                              ├───────────────────────────>│
     │                              │                            │
     │                              │                     거래 완료 상태 변경
```

```json
{
  "eventType": "PAYMENT_COMPLETED",
  "timestamp": "2026-06-13T10:30:00",
  "payload": {
    "auctionId": "A-FNF97",   // auction.auction_id
    "orderId": "ORD-001",
    "winnerId": 42,            // bid.bidder_id (낙찰자)
    "sellerId": 10,            // auction.seller_id
    "paidAmount": 740000,      // auction.current_bid = bid.amount
    "paidAt": "2026-06-13T10:30:00"
  }
}
```

### 2.2 결제 기한 초과 (PAYMENT_TIMEOUT)

| 흐름 | 현재 방식 | 대안 | 대상 | 필요한 데이터 | 이유 |
|------|----------|------|------|-------------|------|
| Payment → Order | Kafka (비동기) | **Kafka + Outbox Pattern** | Order Service | orderId, `auction.auction_id` | 결제 타임아웃 → 주문 취소. Outbox로 원자성 보장 |
| Order → Auction | Kafka (비동기) | **Kafka + Outbox Pattern** | Auction Service | `auction.auction_id`, `bid.bidder_id` (winnerId), `auction.current_bid` | **Order가 중계하여 Auction에 유찰 재처리 전달**. Payment가 Auction을 직접 호출하지 않음 |
| Payment → 프론트 | 없음 (추가 필요) | **Notification Service** (FCM/이메일) | 낙찰자 (제재), 판매자 | `bid.bidder_id`, `auction.seller_id`, `auction.auction_id` | 결제 기한 초과는 사용자가 페이지를 보고 있지 않을 가능성 높음. **푸시/이메일이 적합** |

> **결제 기한 초과 흐름:** Payment → Order → Auction (체인 방식)
> 결제 완료와 동일한 경로로 처리. Payment는 Order에만 타임아웃을 전달하고, Order가 주문 취소 후 Auction에 유찰 재처리를 전파한다.

```
Payment Service                Order Service               Auction Service
     │                              │                            │
     │  PAYMENT_TIMEOUT             │                            │
     │  (orderId)                   │                            │
     ├─────────────────────────────>│                            │
     │                              │  order.status = CANCELLED  │
     │                              │                            │
     │                              │  ORDER_CANCELLED           │
     │                              │  (auctionId, winnerId,     │
     │                              │   reason: PAYMENT_TIMEOUT) │
     │                              ├───────────────────────────>│
     │                              │                            │
     │                              │                     유찰 재처리
     │                              │                     + 제재 요청
```

```json
{
  "eventType": "PAYMENT_TIMEOUT",
  "timestamp": "2026-06-14T15:00:01",
  "payload": {
    "auctionId": "A-FNF97",   // auction.auction_id
    "orderId": "ORD-001",
    "winnerId": 42,            // bid.bidder_id (제재 대상)
    "sellerId": 10,            // auction.seller_id
    "finalBid": 740000         // auction.current_bid
  }
}
```

## 4. Product Service → Auction Service (경매 상품 등록)

경매 상품 등록 시 Product Service가 데이터의 주인(Single Source of Truth)으로서 상품 정보를 저장하고,
Kafka Outbox Pattern으로 Auction Service에 이벤트를 전달하여 경매를 자동 생성한다.

> **설계 원칙:** Auction Service는 Product Service를 REST 호출하지 않는다.
> Product가 Kafka로 이벤트를 발행하고, Auction은 소비만 한다.

### 4.1 흐름

```
프론트                      Product Service                    Auction Service
   │                              │                                  │
   │  POST /api/v1/products       │                                  │
   │  { type: "AUCTION",          │                                  │
   │    name, brand, edition,     │                                  │
   │    category, startPrice,     │                                  │
   │    minIncrement, endsAt,     │                                  │
   │    images }                  │                                  │
   ├─────────────────────────────>│                                  │
   │                              │  @Transactional                  │
   │                              │  ┌───────────────────────┐       │
   │                              │  │ 1. product INSERT     │       │
   │                              │  │    (type=AUCTION)     │ 하나의│
   │                              │  │ 2. outbox INSERT      │ 트랜잭│
   │                              │  │    (PRODUCT_LISTED)   │ 션    │
   │                              │  └───────────────────────┘       │
   │                              │                                  │
   │  201 { productId }           │                                  │
   │<─────────────────────────────│                                  │
   │                              │                                  │
   │                              │  OutboxPoller (1초)               │
   │                              │  └── Kafka: product.listed       │
   │                              │              │                   │
   │                              │              └──────────────────>│
   │                              │                                  │
   │                              │                   auction INSERT │
   │                              │              (이벤트에서 추출)    │
   │                              │                                  │
   │  WebSocket: "경매 등록 완료"  │                                  │
   │<────────────────────────────────────────────────────────────────│
```

### 4.2 Product → Auction 필드 매핑

| product_table 컬럼 | → PRODUCT_LISTED 이벤트 | → auction 컬럼 | 비고 |
|-------------------|----------------------|---------------|------|
| `product_table.name` | `payload.name` | `auction.name` | 상품명 |
| `product_table.brand` (추가) | `payload.brand` | `auction.brand` | 브랜드 |
| `product_table.edition` (추가) | `payload.edition` | `auction.edition` | 에디션 |
| `product_table.category` | `payload.category` | `auction.category` | 카테고리 |
| `product_table.description` | `payload.description` | `auction.description` | 상품 설명 |
| `product_table.user_id` | `payload.sellerId` | `auction.seller_id` | 판매자 회원 ID |
| `product_table.start_price` (추가) | `payload.startPrice` | `auction.start_price` | 경매 시작가 |
| `product_table.min_increment` (추가) | `payload.minIncrement` | `auction.min_increment` | 최소 입찰 단위 |
| `product_table.ends_at` (추가) | `payload.endsAt` | `auction.ends_at` | 경매 종료 시각 |
| `product_image.image_url` (sort_order=0) | `payload.thumbnailUrl` | `auction.thumbnail_url` | 대표 이미지 |

### 4.3 Product 테이블 변경 사항

```sql
-- product_table에 추가 필요한 컬럼 (경매 상품용)
-- product_type은 이미 존재 (GENERAL, AUCTION)

ALTER TABLE product_table ADD COLUMN "brand" VARCHAR(100);
ALTER TABLE product_table ADD COLUMN "edition" VARCHAR(100);
```

### 4.5 원자성 보장 (Outbox Pattern)

| 시나리오 | 결과 |
|---------|------|
| 정상 | Product 저장 + Outbox 저장 (1 트랜잭션) → Kafka 전송 → Auction 생성 |
| Product 서버 다운 | 트랜잭션 롤백 → 상품도 이벤트도 없음 (정합) |
| Auction 서버 다운 | Kafka에 메시지 남음 → 복구 후 자동 consume → 경매 생성 |
| Kafka 전송 실패 | Outbox에 PENDING 상태로 남음 → Poller가 재전송 |

### 4.6 Auction Service 내부 구현 위치 (Clean Architecture)

```
auction/
├── infra/
│   └── kafka/
│       └── ProductListedConsumer.java    ← Kafka Consumer (이벤트 수신)
│           ProductListedPayload.java     ← 이벤트 payload DTO
├── application/
│   └── service/
│       └── AuctionService.java           ← 경매 생성 로직
└── domain/
    └── repository/
        └── AuctionRepository.java        ← 저장
```

```java
// infra/kafka/ProductListedConsumer.java
@Component
@RequiredArgsConstructor
public class ProductListedConsumer {

    private final AuctionService auctionService;

    @KafkaListener(topics = "product.listed", groupId = "auction-service")
    public void consume(ProductListedPayload payload) {
        if (!"AUCTION".equals(payload.productType())) {
            return;  // 일반 상품은 무시
        }
        auctionService.createFromProduct(payload);
    }
}
```

### 4.7 방식 선택 근거

| 항목 | REST (기존) | Kafka + Outbox (채택) |
|------|:----------:|:--------------------:|
| 원자성 | 불완전 (서버 다운 시 불일치) | **완벽 (Outbox 보장)** |
| 결합도 | 강결합 (Auction → Product 의존) | **느슨 (이벤트 소비만)** |
| Product 장애 시 | 경매 등록 불가 | **영향 없음** |
| Auction 장애 시 | 데이터 불일치 | **자동 복구** |
| 데이터 주인 | 모호 | **Product가 주인** |
| REST 호출 | 필요 (FeignClient) | **불필요** |

## 7. 방식 선택 기준

| 기준 | Kafka (비동기) | REST (동기) | WebSocket (실시간) |
|------|:-------------:|:-----------:|:-----------------:|
| 응답 속도 요구 | 수초 허용 | 즉시 응답 필요 | sub-second |
| 결합도 | 느슨 (이벤트 기반) | 강결합 (직접 호출) | 느슨 (구독 기반) |
| 장애 전파 | X (버퍼링) | O (연쇄 장애) | X |
| 재처리 | O (오프셋 리셋) | X (재호출 필요) | X |
| 사용 사례 | 서비스 간 통신 | 동기 조회 필수 시 | 클라이언트 실시간 갱신 |

### 이 프로젝트에서의 선택

| 통신 | 방식 | 이유 |
|------|------|------|
| 서비스 → 서비스 (일반) | **Kafka** | MSA 독립 배포 원칙, 장애 격리, 재처리 가능 |
| 서비스 → 서비스 (돈 관련) | **Kafka + Outbox Pattern** | 이벤트 유실 방지, DB 커밋과 이벤트 발행의 원자성 보장 |
| 서비스 → 서비스 (즉시 실패 감지) | **REST (동기) + Saga 보상** | 결제 준비 등 반드시 성공해야 하는 흐름 |
| 서비스 → 클라이언트 (실시간) | **WebSocket** (+ Redis Pub/Sub) | 입찰 현재가 갱신은 sub-second 필요. 다중 인스턴스 시 Redis 전파 |
| 서비스 → 클라이언트 (일반) | **REST** | 피드 조회, 상세 조회 등 요청-응답 패턴 |
| 서비스 간 데이터 조회 | **Redis 캐시 (Lazy)** | 닉네임 등 변경 빈도 낮은 데이터는 캐시로 충분. Kafka 이벤트 동기화보다 단순 |

### 방식 선택 판단 기준

```
이벤트 유실 시 돈 손실 발생?
    ├── YES → Kafka + Outbox Pattern
    └── NO
         │
         반드시 즉시 성공해야 하는가?
         ├── YES → REST (동기) + Saga 보상
         └── NO
              │
              실시간 클라이언트 갱신 필요?
              ├── YES → WebSocket (+ Redis Pub/Sub)
              └── NO → Kafka (비동기)
```

### Outbox Pattern 구조

돈이 관련된 흐름 (낙찰, 결제, 주문)에서 DB 저장과 이벤트 발행의 원자성을 보장한다.

```
┌─────────────────────────────────────────────────────────────┐
│ Service (예: Payment)                                        │
│                                                              │
│  @Transactional  ─────────────────────────────┐             │
│  │ 1. payment.status = APPROVED               │ 하나의      │
│  │ 2. INSERT INTO outbox_event (              │ 트랜잭션    │
│  │      event_type, payload, status)          │             │
│  │    VALUES ('PAYMENT_COMPLETED',            │             │
│  │      '{"orderId":...}', 'PENDING')         │             │
│  └────────────────────────────────────────────┘             │
│                                                              │
│  OutboxPoller (@Scheduled, 1초 간격)                         │
│  ┌────────────────────────────────────────────┐             │
│  │ SELECT * FROM outbox_event                 │             │
│  │   WHERE status = 'PENDING'                 │             │
│  │                                             │             │
│  │ → kafkaTemplate.send(event)                │             │
│  │ → UPDATE outbox_event SET status = 'SENT'  │             │
│  └────────────────────────────────────────────┘             │
└─────────────────────────────────────────────────────────────┘
```

| 적용 대상 | 서비스 | 이유 |
|-----------|--------|------|
| `AUCTION_ENDED` | Auction | 낙찰 이벤트 유실 → 주문 미생성 → 돈 손실 |
| `PAYMENT_COMPLETED` | Payment | 결제 완료 이벤트 유실 → 주문 상태 불일치 |
| `PAYMENT_TIMEOUT` | Payment | 타임아웃 이벤트 유실 → 유찰 미처리 |
| `ORDER_CREATED` | Order | 주문 이벤트 유실 → 결제 미생성 |

### 8.4 원본 ERD → 현재 구현 변경 이력

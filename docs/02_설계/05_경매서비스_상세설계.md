# Auction Service 상세 설계

## 1. 서비스 개요

| 항목 | 내용 |
|------|------|
| 서비스명 | auction-service |
| 포트 | 8084 |
| 아키텍처 | **Clean Architecture** (도메인별 4-Layer: presentation → application → domain ← infra) |
| DB | biddy_auction (PostgreSQL 16) |
| 캐시/Pub-Sub | Redis 7 |
| 통신 | REST (동기), Kafka (비동기), WebSocket (실시간) |
| Java | 21, Spring Boot 3.4, Spring Cloud 2024 |

경매 상태 관리, 입찰, 관심 경매, 실시간 입찰 구독 기능을 담당하는 도메인 서비스.
Auction, Bid, Watch 세 개의 독립 도메인으로 구성되며, 도메인 간 참조는 ID 기반으로 느슨하게 연관된다.

**설계 원칙: Auction은 경매 상태만 관리, 상품 정보는 Product Service 책임**
- Auction 테이블에 `productId`만 보관 (상품명, 브랜드, 이미지 등 미보관)
- 경매 피드/상세 화면은 **API Gateway/BFF에서 Product + Auction 응답을 조합**
- Product Service 장애 시 경매 상태(입찰/종료)는 독립 동작, 상품 정보만 미표시

      "startPrice": 400000,
      "minIncrement": 20000,
      "currentBid": 720000,
      "bidCount": 6,
      "endsAt": "2026-06-12T15:30:00",

>   Auction 데이터 + Product 데이터 merge

> ```

```json
{
  "auctionId": "A-FNF97",
  "status": "ENDED",
  "currentBid": 0,
  "result": {
```

> `result.type`: `SOLD` (낙찰) | `UNSOLD` (유찰)
> `result.paymentStatus`: `PENDING` | `PAID` | `TIMEOUT` (낙찰 시에만)

### 2.3 경매 등록 (Product Service → Kafka → Auction Service)

> Auction Service에 직접 POST API가 아닌, **Product Service에서 경매 상품 등록 시
> Kafka로 이벤트를 발행하여 Auction Service가 자동 생성**한다.

```
프론트 → POST /api/v1/products (Product Service)
         { type: "AUCTION", name, brand, edition, category,
           startPrice, minIncrement, endsAt, images }

Product Service:
  @Transactional
  1. product INSERT (type=AUCTION)
  2. Kafka 발행 (product.auction.registered)

  → Kafka payload: { productId, sellerId, startPrice, minIncrement, startsAt, endsAt }
  → Auction Service Consumer → auction INSERT (productId + 경매 설정만 저장)
```

**도메인 책임 분리**
- **Product Service 책임**: 상품명, 브랜드, 에디션, 카테고리, 설명, 이미지, 판매자 정보
- **Auction Service 책임**: productId 참조 + 경매 상태(가격, 입찰, 시간, 낙찰자)
- Auction 테이블에 상품 정보(name, brand 등)를 저장하지 않음
- 경매 피드/상세 화면은 **API Gateway/BFF에서 조합**

상세 설계: `06_도메인간_이벤트_흐름_정의.md` 참조

### 2.11 경매 종료 후 전체 API 흐름

```
경매 종료 시각 도달
    │
    ▼
[Internal] AuctionCloseScheduler
    │
    ├── 입찰 있음 ──────────────────────────────────────┐
    │   → status = ENDED                                │
    │   → Kafka: auction.ended (SOLD)                   │
    │   → WebSocket: ENDED push                         │
    │                                                   ▼
    │                                    ┌──────────────────────────┐
    │                                    │ 낙찰자: 결제 기한 24시간  │
    │                                    └──────────┬───────────────┘
    │                                               │
    │                                    ┌──────────┴──────────┐
    │                                    │                     │
    │                                결제 완료              기한 초과
    │                                    │                     │
    │                                    ▼                     ▼
    │                            주문/결제 완료         입찰자 제재
    │                            (Order/Payment)       + 유찰 재처리
    │                                                  (Kafka: payment.timeout)
    │
    └── 입찰 없음
        → status = ENDED
        → Kafka: auction.ended (UNSOLD)
        → 판매자 알림: "유찰되었습니다"
```

**경매 종료 후 사용되는 API/이벤트**

| 순서 | 주체 | API / 이벤트 | 설명 |
|------|------|-------------|------|
| 1 | Scheduler | (Internal) | `ends_at <= NOW()` 경매 종료 처리 |
| 2 | Auction → Kafka | `auction.ended` | 낙찰/유찰 이벤트 발행 |
| 3 | Auction → Client | WebSocket ENDED | 실시간 종료 알림 |
| 4 | Client → Auction | `GET /api/v1/auctions/{id}` | 종료된 경매 상세 (result 포함) |
| 5 | Client → Auction | `GET /api/v1/auctions/{id}/result` | 낙찰 결과 전용 조회 |
| 6 | Order → Auction | Kafka consume `auction.ended` | 낙찰 주문 자동 생성 |
| 7 | Payment → Order | 결제 처리 | 24시간 내 결제 |
| 8 | Payment → Kafka | `payment.completed` / `payment.timeout` | 결제 결과 이벤트 |
| 9 | Auction ← Kafka | consume `payment.timeout` | 유찰 재처리 + 제재 |

## 3. ERD

### 3.1 auction 테이블

> **Auction은 경매 상태만 관리. 상품 정보(name, brand 등)는 Product Service 책임.**

| 컬럼 | 타입 | 제약조건 | 설명 |
|------|------|----------|------|
| auction_id | VARCHAR(20) | PK | 경매 ID (e.g. A-FNF97) |
| product_id | UUID | NOT NULL, UNIQUE | Product Service 상품 ID (참조용) |
| seller_id | BIGINT | NOT NULL | 판매자 회원 ID |
| start_price | BIGINT | NOT NULL | 시작 가격 |
| min_increment | BIGINT | NOT NULL | 최소 입찰 단위 |
| current_bid | BIGINT | NOT NULL, DEFAULT 0 | 현재 최고 입찰가 |
| bid_count | INT | NOT NULL, DEFAULT 0 | 총 입찰 횟수 |
| watcher_count | INT | NOT NULL, DEFAULT 0 | 관심 등록 수 |
| status | VARCHAR(10) | NOT NULL, DEFAULT 'LIVE' | LIVE, ENDED |
| starts_at | TIMESTAMP | | 경매 시작 시각 |
| ends_at | TIMESTAMP | NOT NULL | 경매 종료 시각 |
| winner_id | BIGINT | | 낙찰자 회원 ID (종료 후 확정) |
| winning_bid_id | BIGINT | | 낙찰 입찰 ID (종료 후 확정) |
| created_at | TIMESTAMP | NOT NULL, DEFAULT now() | 등록 시각 |
| updated_at | TIMESTAMP | NOT NULL, DEFAULT now() | 수정 시각 |

**제거된 컬럼 (v1 대비)**

| 컬럼 | 이유 |
|------|------|
| `name`, `brand`, `edition` | Product Service 책임 → 비정규화 제거 |
| `category`, `description` | Product Service 책임 |
| `thumbnail_url` | Product Service 책임 |

**추가된 컬럼 (v1 대비)**

| 컬럼 | 이유 |
|------|------|
| `product_id` | Product 참조 (Gateway 조합 키) |
| `starts_at` | 예약 경매 지원 |
| `winner_id` | 낙찰 결과 즉시 조회 (bid 테이블 재조회 불필요) |
| `winning_bid_id` | 낙찰 입찰 추적 |

### 3.2 bid 테이블

| 컬럼 | 타입 | 제약조건 | 설명 |
|------|------|----------|------|
| bid_id | BIGSERIAL | PK | 입찰 ID |
| auction_id | VARCHAR(20) | NOT NULL, FK → auction | 경매 ID |
| bidder_id | BIGINT | NOT NULL, FK(외부) | 입찰자 회원 ID |
| amount | BIGINT | NOT NULL | 입찰 금액 |
| bid_at | TIMESTAMP | NOT NULL, DEFAULT now() | 입찰 시각 |

- INDEX: `(auction_id, bid_at DESC)` — 입찰 내역 최신순 조회용
- INDEX: `(auction_id, amount DESC)` — 최고 입찰 조회용

### 3.3 auction_watch 테이블

| 컬럼 | 타입 | 제약조건 | 설명 |
|------|------|----------|------|
| watch_id | BIGSERIAL | PK | 관심 ID |
| auction_id | VARCHAR(20) | NOT NULL, FK → auction | 경매 ID |
| member_id | BIGINT | NOT NULL, FK(외부) | 회원 ID |
| created_at | TIMESTAMP | NOT NULL, DEFAULT now() | 등록 시각 |

- UNIQUE: `(auction_id, member_id)` — 중복 관심 등록 방지

### 3.4 설계 변경 노트

> v1에서는 `auction_image` 테이블이 있었으나, 이미지는 Product Service 책임이므로 제거.
> v1에서 비정규화 저장하던 상품 정보(name, brand 등)도 제거. `productId`만 보관.

```
└──────────────────┘
```

> `auction_image` 테이블 제거 — 이미지는 Product Service 책임

## 4. Kafka 이벤트

### 4.1 Produce (발행)

| 토픽 | 이벤트 | 트리거 | 설명 |
|------|--------|--------|------|
| `auction.ended` | AuctionEndedEvent | 경매 종료 시 (스케줄러) | 낙찰 결과를 Order에 전달 (주문 자동 생성) |

> 입찰 알림(`auction.bid`)은 Kafka가 아닌 **WebSocket 직접 push**로 처리.
> Auction 응답에 상품 정보가 없으므로, Order가 주문 생성 시 Product Service에서 상품 정보를 별도 조회.

**AuctionEndedEvent** (낙찰 → Order Service)
```json
{
  "eventType": "AUCTION_ENDED",
  "timestamp": "2026-06-12T15:30:00",
  "auctionId": "A-FNF97",
  "productId": "uuid-...",
  "sellerId": 10,
  "finalBid": 740000,
  "winnerId": 42,
  "paymentDeadline": "2026-06-13T15:30:00"
}
```

> 상품명, 브랜드, 썸네일 등은 포함하지 않음.
> Order Service가 `productId`로 Product Service에서 조회하여 주문 내역에 표시.

### 4.2 Consume (구독)

| 토픽 | 이벤트 | 처리 |
|------|--------|------|
| `product.auction.registered` | ProductAuctionRegisteredEvent | 경매 자동 생성 (productId + 경매 설정만 저장) |

> 결제 관련(`payment.completed`, `payment.timeout`)은 **Order/Payment 서비스 간 흐름**.
> Auction Service 범위 아님.

**ProductAuctionRegisteredEvent** (수신)
```json
{
  "productId": "uuid-...",
  "sellerId": 10,
  "startPrice": 400000,
  "minIncrement": 20000,
  "startsAt": "2026-06-20T10:00:00",
  "endsAt": "2026-06-20T15:00:00"
}
```

> 상품 정보(name, brand 등)는 포함하지 않음. Auction은 `productId`만 저장.

### 4.3 Auction Kafka 현황 (확정)

| 방향 | Topic | 역할 | Payload 핵심 필드 |
|------|-------|------|------------------|
| Product → **Auction** | `product.auction.registered` | 경매 자동 생성 | productId, sellerId, startPrice, minIncrement, endsAt |
| Auction → **Order** | `auction.ended` | 낙찰 주문 생성 | auctionId, productId, sellerId, finalBid, winnerId |

> 입찰 알림은 WebSocket 직접 push (Kafka 불필요).
> 결제 관련 이벤트는 Order/Payment 간 흐름 (Auction 범위 아님).

#### 이벤트 전체 흐름도

```
┌──────────────────────────────────────────────────────────┐
│                    Kafka Event Bus                         │
│                                                            │
│  product.auction.registered ──────┐                       │
│                                    ▼                       │
│                                 Auction                    │
│                              (경매 자동 생성)               │
│                                                            │
│  auction.ended ───────────────────┐                       │
│                                    ▼                       │
│                                 Order                      │
│                              (주문 자동 생성)               │
│                                                            │
│  입찰 알림 ──── WebSocket 직접 push → 프론트 (실시간)      │
│  종료 알림 ──── WebSocket 직접 push → 프론트 (실시간)      │
└──────────────────────────────────────────────────────────┘
```

#### 이벤트 설계 원칙

| 원칙 | 이유 |
|------|------|
| **ID 필드만 포함 (도메인 순수성)** | 상품 정보는 Product 책임, 필요 시 소비자가 Product API 호출 |
| **timestamp 필수** | 이벤트 순서 보장, 디버깅 |
| **eventType 필수** | 하나의 topic에서 여러 이벤트 타입 분기 가능 |
| **민감 정보 절대 제외** | 비밀번호, 카드번호, 주소 등 포함 금지 |

## 5. 핵심 비즈니스 규칙

### 5.1 입찰 검증

```
입찰 금액 >= currentBid + minIncrement
```

| 검증 항목 | 실패 시 응답 |
|-----------|-------------|
| 경매가 LIVE 상태인지 | 409 Conflict — 종료된 경매 |
| 입찰 금액 >= currentBid + minIncrement | 400 Bad Request — 최소 입찰 단위 미달 |
| 현재 최고 입찰보다 높은지 (동시성) | 409 Conflict — 이미 더 높은 입찰 존재 |
| 본인 경매에 입찰 불가 | 403 Forbidden |

### 5.2 동시성 제어

입찰은 동시 요청이 빈번하므로 **비관적 잠금(Pessimistic Lock)** 또는 **낙관적 잠금(Optimistic Lock + 재시도)** 적용.

```java
// 비관적 잠금 예시
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT a FROM Auction a WHERE a.auctionId = :id")
Optional<Auction> findByIdForUpdate(@Param("id") String id);
```

### 5.3 경매 종료 처리

- **스케줄러**: `@Scheduled`로 주기적 만료 경매 확인 또는 각 경매별 `endsAt`에 맞춘 지연 작업
- 종료 시: `status`를 `ENDED`로 변경 → `auction.ended` 이벤트 발행
- 후속 처리: Order Service에서 낙찰 주문 자동 생성

## 6. 입찰 플로우 상세 설계

플로우차트 기반으로 입찰 진입 → 입찰 처리 → 경매 종료까지의 전체 흐름을 정의한다.

### 6.1 입찰 진입 플로우

사용자가 경매 상세 페이지에 진입한 시점부터 입찰 금액을 입력하기까지의 사전 검증 단계.

```
경매 상세 진입
    │
    ▼
경매 상세 페이지 렌더링
    │
    ▼
┌─────────────┐
│ 경매 상태?  │
└─────┬───────┘
      │
  ┌───┴───┐
  │종료   │진행중
  ▼       ▼
낙찰     로그인 여부 확인
프로세스      │
         ┌───┴───┐
         │아니요  │예
         ▼       ▼
     로그인    판매자 본인 여부
     화면유도      │
              ┌───┴───┐
              │예     │아니요
              ▼       ▼
          본인입찰  입찰 금액 입력
          불가안내
```

| 단계 | 검증 항목 | 실패 시 처리 | HTTP 상태 |
|------|-----------|-------------|-----------|
| 1 | 경매 상태 = LIVE | 종료된 경매 → 낙찰 프로세스로 라우팅 | 409 Conflict |
| 2 | 사용자 인증 여부 | 비로그인 → 로그인 화면 유도 (FE 처리) | 401 Unauthorized |
| 3 | 판매자 본인 여부 | sellerId == currentUserId → 입찰 거부 | 403 Forbidden |

### 6.2 입찰 처리 플로우

입찰 금액 제출 후 서버 내부의 검증-락-저장-알림 파이프라인.

```
입찰 금액 입력
    │
    ▼
┌───────────────┐
│ 1차 유효 검증 │ ← Application Layer (락 없이 빠른 검증)
└───────┬───────┘
        │ 통과
        ▼
  비관적 락 획득 ← SELECT ... FOR UPDATE
        │
        ▼
┌───────────────┐
│ 최종 유효 검증 │ ← 락 보유 상태에서 재검증
└───────┬───────┘
        │ 검증 통과
        ▼
  입찰 완료 (현재가 업데이트)
        │
        ├──────────────────┐
        ▼                  ▼
  이전 입찰자 알림    경쟁 패배 안내
  (WebSocket/Push)   (경쟁가 갱신 알림)
        │                  │
        └──────┬───────────┘
               ▼
       입찰 금액 입력 (재입찰 가능)
```

#### 6.2.1 1차 유효 검증 (Pre-Lock Validation)

락 획득 전 빠르게 수행하는 사전 검증. DB 락 대기 시간을 줄이기 위한 목적.

| 검증 항목 | 조건 | 실패 응답 |
|-----------|------|-----------|
| 경매 상태 | status == LIVE | 409 — 종료된 경매 |
| 최소 입찰 단위 | amount >= currentBid + minIncrement | 400 — 최소 입찰 단위 미달 |
| 본인 경매 여부 | bidderId != sellerId | 403 — 본인 입찰 불가 |

#### 6.2.2 비관적 락 획득 (Pessimistic Lock)

```java
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT a FROM Auction a WHERE a.auctionId = :id")
Optional<Auction> findByIdForUpdate(@Param("id") String id);
```

- 동시 입찰 요청에 대한 직렬화 보장
- 락 대기 타임아웃 설정 필요 (e.g. `javax.persistence.lock.timeout=3000`)
- 락 획득 실패 시 → 503 Service Unavailable 또는 재시도 안내

#### 6.2.3 최종 유효 검증 (Post-Lock Validation)

락을 잡은 상태에서 **최신 DB 값** 기반으로 재검증. 1차 검증과 락 획득 사이에 다른 트랜잭션이 값을 변경했을 수 있으므로 반드시 필요.

| 검증 항목 | 조건 | 실패 응답 |
|-----------|------|-----------|
| 경매 상태 (재확인) | status == LIVE | 409 — 종료된 경매 |
| 입찰 금액 (재확인) | amount >= currentBid + minIncrement | 409 — 이미 더 높은 입찰 존재 |

#### 6.2.4 입찰 완료 처리

하나의 트랜잭션 내에서 원자적으로 수행:

1. `bid` 레코드 INSERT — `bidRepository.save()`
2. `auction.current_bid` UPDATE — `auction.applyBid(amount)`
3. `auction.bid_count` INCREMENT — `auction.applyBid()` 내부
4. WebSocket 실시간 브로드캐스트 — `AuctionWebSocketPublisher.publishBid()`
5. (TODO) Kafka `auction.bid` 토픽 발행 — `BidPlacedEvent`

> 현재 구현: 1~4 완료. Kafka 발행은 다음 작업에서 추가 예정.

#### 6.2.5 이전 입찰자 알림

입찰 성공 시, **직전 최고 입찰자**에게 경쟁 패배 알림 전송.

| 알림 대상 | 채널 | 내용 | 구현 |
|-----------|------|------|------|
| 경매 구독자 전체 (이전 입찰자 포함) | WebSocket (`/topic/auctions/{auctionId}`) | BID 메시지 (currentBid, bidCount, bidderId) | `AuctionWebSocketPublisher.publishBid()` |

> Member Service 경유 없이 WebSocket으로 직접 push. 경매 페이지를 구독 중인 모든 사용자에게 전달.

### 6.3 경매 종료 플로우

경매 종료 시각 도달 시 스케줄러에 의해 자동 실행되는 낙찰/유찰 처리 흐름.

```
종료 시각 도달
    │
    ▼
종료 정보 수집
    │
    ▼
┌─────────────────┐
│ 입찰 내역 존재? │
└────┬────────────┘
     │
 ┌───┴───┐
 │없음   │있음
 ▼       ▼
유찰    최고 입찰자 낙찰
(미판매)     │
             ▼
       낙찰 알람 발송
             │
             ▼
     ┌──────────────────┐
     │ 24시간 이내 결제? │
     └────┬─────────────┘
          │
      ┌───┴────┐
      │아니요   │예
      ▼        ▼
  입찰자 제재  주문/결제 완료
  + 유찰처리   흐름
```

#### 6.3.1 종료 정보 수집

스케줄러가 `ends_at <= NOW()` AND `status = 'LIVE'`인 경매를 조회하여 종료 처리 대상 수집.

```java
@Scheduled(fixedDelay = 1000)
public void processExpiredAuctions() {
    List<Auction> expired = auctionRepository
        .findAllByStatusAndEndsAtBefore(AuctionStatus.LIVE, LocalDateTime.now());
    expired.forEach(this::closeAuction);
}
```

#### 6.3.2 유찰 처리 (입찰 내역 없음)

| 처리 항목 | 내용 |
|-----------|------|
| 상태 변경 | status → `ENDED` |
| 결과 기록 | 낙찰자 없음 (winnerId = null) |
| 이벤트 발행 | `AuctionEndedEvent` (winnerId = null) |
| WebSocket | UNSOLD 이벤트 브로드캐스트 (`AuctionWebSocketPublisher.publishUnsold()`) — 판매자가 경매 페이지 구독 중이면 직접 알림 |

#### 6.3.3 낙찰 처리 (입찰 내역 존재)

| 처리 항목 | 내용 |
|-----------|------|
| 상태 변경 | status → `ENDED` |
| 낙찰자 확정 | 최고 입찰 금액의 bidder를 winner로 확정 |
| 이벤트 발행 | `AuctionEndedEvent` (winnerId, finalBid 포함) |
| WebSocket | ENDED 이벤트 브로드캐스트 (`AuctionWebSocketPublisher.publishEnded()`) — 낙찰자/판매자가 경매 페이지 구독 중이면 직접 알림 |

#### 6.3.4 결제 기한 정책

| 항목 | 정책 |
|------|------|
| 결제 기한 | 낙찰 후 24시간 |
| 기한 내 결제 | Order Service → Payment Service 결제 완료 흐름 |
| 기한 초과 | 입찰자 제재 + 유찰 재처리 |

#### 6.3.5 결제 기한 초과 시 처리

| 처리 항목 | 내용 |
|-----------|------|
| 입찰자 제재 | 패널티 부과 (제재 이력 기록, 반복 시 이용 제한) |
| 유찰 재처리 | 해당 경매를 미판매 상태로 전환 |
| 판매자 알림 | "낙찰자가 결제를 완료하지 않아 유찰 처리되었습니다." |
| 이벤트 흐름 | Payment → Kafka → **Order** (주문 취소) → Kafka → **Auction** (유찰 재처리). Payment가 Auction을 직접 호출하지 않음 |

## 7. 패키지 구조 (Clean Architecture)

도메인별로 독립된 패키지를 가지며, 의존성 방향은 항상 **바깥 → 안쪽**으로 흐른다.

```
com.biddy.auction
├── AuctionApplication.java
│
├── common/                                    ── 공통 모듈 ──
│   ├── config/
│   │   └── JpaConfig.java                     @EnableJpaAuditing 전역 설정
│   ├── domain/
│   │   └── BaseEntity.java                    createdAt, updatedAt 자동 관리
│   └── exception/
│       ├── ErrorCode.java                     에러 코드 Enum (AUCTION_NOT_FOUND 등)
│       ├── BusinessException.java             비즈니스 예외 (RuntimeException)
│       ├── ErrorResponse.java                 API 에러 응답 DTO
│       └── GlobalExceptionHandler.java        @RestControllerAdvice 전역 예외 처리
│
├── auction/                                   ── 경매 도메인 ──
│   ├── application/                           [Application Layer]
```

### 7.1 레이어별 의존성 규칙

```
┌─────────────────────────────────────────────────────┐
│ presentation  →  application  →  domain  ←  infra   │
│ (Controller)     (Service)      (Entity)   (JPA)    │
│                  (UseCase)      (Port)     (Adapter) │
└─────────────────────────────────────────────────────┘
         ↓              ↓            ↑          ↑
    외부 요청 수신   비즈니스 로직   핵심 도메인   기술 구현
```

| 규칙 | 설명 |
|------|------|
| **Domain은 외부에 의존하지 않는다** | Entity, Repository Port는 순수 Java |
| **Application은 Domain만 의존** | UseCase/Service는 Repository Port만 참조 |
| **Infra는 Domain을 구현** | Adapter가 Repository Port를 implements |
| **Presentation은 Application만 의존** | Controller → UseCase 인터페이스 참조 |
| **도메인 간 참조는 ID 기반** | Bid → Auction 직접 참조 X, auctionId로 느슨한 연관 |

### 7.2 구 패키지 → 신 패키지 매핑

| 구 패키지 | 신 패키지 | 변경 이유 |
|-----------|-----------|-----------|
| `controller/` | `{domain}/presentation/` | 도메인별 분리 + Clean Architecture 레이어 명칭 |
| `dto/request/`, `dto/response/` | `application/dto/`, `presentation/dto/` | Input/Output DTO를 레이어별 분리 |
| `entity/` | `{domain}/domain/model/` | 도메인 모델 레이어 명확화 |
| `repository/` | `{domain}/domain/repository/` (Port) + `{domain}/infra/persistence/` (Adapter) | DIP 적용 (Port-Adapter 분리) |
| `service/` | `{domain}/application/service/` + `usecase/` | UseCase 인터페이스 분리 |
| `config/` | `common/config/` | 전역 설정 공통화 |
| `exception/` | `common/exception/` | 전역 예외 처리 공통화 |

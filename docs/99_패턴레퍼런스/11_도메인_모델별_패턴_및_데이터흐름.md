# 도메인 모델별 패턴 및 데이터 흐름

> Auction Service의 각 도메인 모델이 어떤 패턴을 사용하고, 데이터가 어떻게 흘러가는지 정리한다.

---

## 1. 모델 전체 관계도

```
┌─────────────────────────────────────────────────────────────────────┐
│                        Auction Service                              │
│                                                                     │
│  ┌─────────────┐     1:N      ┌─────────────┐                      │
│  │   Auction    │<────────────│     Bid      │                      │
│  │  (경매)      │  auctionId  │   (입찰)     │                      │
│  └──────┬──────┘              └─────────────┘                      │
│         │                                                           │
│         │ 1:N                                                       │
│         │ auctionId                                                 │
│  ┌──────▼──────┐                                                    │
│  │ AuctionWatch │                                                    │
│  │  (관심)      │                                                    │
│  └─────────────┘                                                    │
│                                                                     │
│  ┌─────────────┐                                                    │
│  │AuctionStatus│  LIVE | ENDED (Enum, 별도 테이블 없음)              │
│  └─────────────┘                                                    │
└─────────────────────────────────────────────────────────────────────┘
```

### 연관 방식: ID 참조 (FK 없음)

```
Auction.auctionId  <──  Bid.auctionId         (ID 참조, @ManyToOne 아님)
Auction.auctionId  <──  AuctionWatch.auctionId (ID 참조, @ManyToOne 아님)
Auction.sellerId   <──  Member Service         (외부 서비스, ID만 보관)
Bid.bidderId       <──  Member Service         (외부 서비스, ID만 보관)
```

> MSA 환경에서 JPA 연관관계 대신 ID 참조를 사용하여 도메인 간 결합도를 낮춘다.

## 2. Auction 모델 (경매)

### 2.1 역할

```
경매의 생명주기를 관리하는 핵심 Aggregate Root.
생성(Kafka) -> 진행(LIVE) -> 입찰 적용 -> 종료(ENDED)
```

### 2.2 사용 패턴

```
┌──────────────────────────────────────────────────────────────────┐
│                     Auction 모델 적용 패턴                        │
│                                                                  │
│  1. Rich Domain Model         도메인 메서드로 비즈니스 로직 캡슐화│
│  2. Aggregate Root            Bid, Watch의 상태 변경을 Auction 관리│
│  3. DIP (의존성 역전)          domain -> Port <- infra (Adapter) │
│  4. Pessimistic Lock          입찰 시 SELECT FOR UPDATE          │
│  5. Denormalization           Product 데이터 비정규화 복사        │
│  6. Event-Driven Creation     Kafka Consumer로 자동 생성         │
│  7. BaseEntity 상속           createdAt, updatedAt 자동 관리     │
└──────────────────────────────────────────────────────────────────┘
```

### 2.3 도메인 메서드

```java
Auction
  isLive()                 // 상태 확인 (LIVE 여부)
  applyBid(amount)         // 입찰 적용 (currentBid 갱신 + bidCount++)
  close()                  // 경매 종료 (status -> ENDED)
  hasBids()                // 입찰 존재 여부 (bidCount > 0)
  incrementWatcherCount()  // 관심 등록 수 +1
  decrementWatcherCount()  // 관심 등록 수 -1 (0 미만 방지)
```

### 2.4 데이터 흐름

```
[생성]
  Product Service --Kafka--> ProductAuctionRegisteredConsumer
                                    |
                                    v
                            AuctionService.createFromProduct()
                                    |
                                    v
                            AuctionRepository.save(auction)
                                    |
                                    v
                                 auction 테이블 INSERT

[조회 - 피드]
  Client --GET /auctions--> AuctionController.getAuctionFeed()
                                    |
                                    v
                            AuctionService.getAuctionFeed(query)
                                    |
                                    v
                            AuctionRepository.findByFilters(status, category, pageable)
                                    |
                                    v
                            AuctionJpaRepository --JPQL--> DB
                                    |
                                    v
                            Auction Entity -> AuctionFeedResult -> AuctionFeedResponse
                                    |
                                    v
                                 Client

[조회 - 상세]
  Client --GET /auctions/{id}--> AuctionController.getAuctionDetail()
                                    |
                                    v
                            AuctionService.getAuctionDetail(id)
                                    |
                                    +-- AuctionRepository.findById(id)
                                    +-- BidRepository.findTopByAuctionId(id)  <- topBidder
                                    |
                                    v
                            AuctionDetailResult -> AuctionDetailResponse
                                    |
                                    v
                                 Client

[입찰 적용 - Auction 관점]
  BidService.placeBid()
        |
        +-- AuctionRepository.findById()           <- 1차 검증 (락 없이)
        +-- AuctionRepository.findByIdForUpdate()   <- 비관적 락
        |
        v
  lockedAuction.applyBid(amount)                   <- 도메인 메서드
        |  currentBid = amount
        |  bidCount++
        v
  @Transactional 종료 시 Dirty Checking -> DB UPDATE

[종료]
  AuctionCloseScheduler (1초 간격)
        |
        v
  AuctionRepository.findExpiredLiveAuctions()
        |
        v
  auction.close()                                  <- 도메인 메서드
        |  status = ENDED
        |
        +-- hasBids() == true  -> WebSocket ENDED (winnerId, finalBid)
        +-- hasBids() == false -> WebSocket UNSOLD (auctionId)
        |
        v
  @Transactional 종료 시 Dirty Checking -> DB UPDATE
```

## 3. Bid 모델 (입찰)

### 3.1 역할

```
하나의 경매에 대한 개별 입찰 기록.
생성 후 변경되지 않는 불변(Immutable) 기록.
```

### 3.2 사용 패턴

```
┌──────────────────────────────────────────────────────────────────┐
│                       Bid 모델 적용 패턴                          │
│                                                                  │
│  1. Immutable Record          생성 후 수정 불가 (INSERT only)    │
│  2. ID 참조 (느슨한 연관)      Auction과 FK 없이 auctionId로 참조│
│  3. Pessimistic Lock 대상      입찰 시 Auction을 락 -> Bid 생성  │
│  4. 2단계 검증                 Pre-Lock -> Post-Lock 분리 검증   │
│  5. Event 발행 트리거          입찰 성공 -> WebSocket BID push   │
│  6. @PrePersist               bidAt 자동 설정                   │
│  7. 복합 인덱스               (auction_id, bid_at DESC)         │
│                               (auction_id, amount DESC)         │
└──────────────────────────────────────────────────────────────────┘
```

### 3.3 데이터 흐름

```
[입찰 실행]
  Client --POST /auctions/{id}/bids--> BidController.placeBid()
                                           |
                                           v
                                    BidService.placeBid(command)
                                           |
    +--------------------------------------+
    |                                      |
    |  (1) 1차 유효 검증 (Pre-Lock)        |
    |     Auction 조회 (락 없이)           |
    |     +-- isLive() 확인                |
    |     +-- 본인 경매 확인               |
    |     +-- 금액 검증                    |
    |                                      |
    |  (2) 비관적 락 획득                  |
    |     SELECT ... FOR UPDATE            |
    |                                      |
    |  (3) 최종 유효 검증 (Post-Lock)      |
    |     +-- isLive() 재확인              |  <- 1차와 락 사이에 다른 TX가
    |     +-- 금액 재검증                  |     값을 변경했을 수 있음
    |                                      |
    |  (4) Bid 저장                        |
    |     bidRepository.save(bid)          |
    |                                      |
    |  (5) Auction 현재가 갱신             |
    |     auction.applyBid(amount)         |
    |                                      |
    |  (6) WebSocket push                  |
    |     publishBid(auctionId, ...)       |
    |                                      |
    +--------------------------------------+
                                           |
                                    PlaceBidResult -> PlaceBidResponse
                                           |
                                           v
                                        Client

[입찰 내역 조회]
  Client --GET /auctions/{id}/bids--> BidController.getBidHistory()
                                           |
                                           v
                                    BidService.getBidHistory(query)
                                           |
                                           v
                                    BidRepository.findByAuctionId(id, pageable)
                                           |  Sort: bidAt DESC (최신순)
                                           v
                                    Bid Entity -> BidHistoryResult -> BidHistoryResponse

[내 입찰 참여 목록]
  Client --GET /members/me/bids--> MyBidController.getMyBids()
                                           |
                                           v
                                    BidService.getMyBids(bidderId, status, page, size)
                                           |
                                           +-- findDistinctAuctionIdsByBidderId()
                                           |      -> auctionId 목록
                                           +-- auctionRepository.findById() x N
                                           +-- findTopByAuctionIdAndBidderId()  <- myHighestBid
                                           +-- findTopByAuctionId()             <- isTopBidder
                                           |
                                           v
                                    MyBidResult -> MyBidResponse
```

## 4. AuctionWatch 모델 (관심 경매)

### 4.1 역할

```
회원이 특정 경매를 관심 등록한 기록.
토글 방식: 등록 <-> 해제.
Auction의 watcherCount와 동기화.
```

### 4.2 사용 패턴

```
┌──────────────────────────────────────────────────────────────────┐
│                   AuctionWatch 모델 적용 패턴                     │
│                                                                  │
│  1. Toggle Pattern            동일 요청 -> 등록/해제 교차        │
│  2. Unique Constraint         (auction_id, member_id) UK         │
│  3. 연관 Entity 상태 동기화    Watch 등록/해제 시 watcherCount 갱신│
│  4. @PrePersist               createdAt 자동 설정                │
│  5. ID 참조                   Auction, Member와 FK 없이 ID 참조  │
└──────────────────────────────────────────────────────────────────┘
```

### 4.3 데이터 흐름

```
[관심 등록/해제 토글]
  Client --POST /auctions/{id}/watch--> WatchController.toggleWatch()
                                             |
                                             v
                                      WatchService.toggleWatch(auctionId, memberId)
                                             |
                                             v
                                      AuctionRepository.findById(auctionId)
                                             |
                              +--------------+--------------+
                              |                             |
                    Watch 존재?                    Watch 미존재?
                              |                             |
                              v                             v
                    watchRepository.delete()     watchRepository.save()
                    auction.decrementWatcher()   auction.incrementWatcher()
                              |                             |
                              v                             v
                    { watching: false,           { watching: true,
                      watcherCount: N-1 }         watcherCount: N+1 }

[내 관심 목록]
  Client --GET /members/me/watches--> MyWatchController.getMyWatches()
                                             |
                                             v
                                      WatchService.getMyWatches(memberId, page, size)
                                             |
                                             v
                                      WatchRepository.findByMemberId(memberId, pageable)
                                             |
                                             +-- AuctionWatch.auctionId 추출
                                             +-- AuctionRepository.findById() x N
                                             |
                                             v
                                      MyWatchResult -> MyWatchResponse
```

## 5. AuctionStatus 열거형 (Enum)

### 5.1 상태 전이

```
        Kafka Consumer            @Scheduled
            (생성)                 (종료 처리)
              |                       |
              v                       v
           +------+               +------+
           | LIVE |-------------->|ENDED |
           +------+               +------+
              |                       |
              |  입찰 가능            |  입찰 불가
              |  관심 등록 가능       |  결과 조회 가능
              |  WebSocket 구독      |  WebSocket ENDED/UNSOLD push
              |                       |
              +-- 스케줄러가          +-- 낙찰 (SOLD) 또는
                  endsAt 경과 시          유찰 (UNSOLD)
                  자동 전이
```

### 5.2 상태별 API 접근 제어

| API | LIVE | ENDED |
|-----|:----:|:-----:|
| 피드 조회 | O | O |
| 상세 조회 | O | O |
| 입찰 | O | X (409) |
| 입찰 내역 | O | O |
| 관심 등록 | O | O |
| 낙찰 결과 | X (409 AUCTION_STILL_LIVE) | O (SOLD/UNSOLD) |
| WebSocket | BID 메시지 수신 | ENDED/UNSOLD 수신 |

## 6. 모델 간 상호작용 종합

```
Product Service --Kafka--> [Auction 생성]
                                |
                +---------------+---------------+
                |               |               |
                v               v               v
           입찰 흐름        종료 흐름       관심 흐름
                |               |               |
                v               v               v
         +---------+    +-------------+   +-----------+
         |   Bid   |    |  Scheduler  |   |  Watch    |
         |  생성   |    |  경매 종료  |   |  토글     |
         +----+----+    +------+------+   +-----+-----+
              |                |                |
              v                v                v
     +-----------------------------------------------+
     |              Auction (Aggregate Root)           |
     |                                                 |
     |  applyBid()     close()     increment/          |
     |  currentBid++   ENDED       decrement           |
     |  bidCount++                 WatcherCount         |
     +------------------------+------------------------+
                              |
                              v
                +-----------------------+
                |   WebSocket Push      |
                |                       |
                |  BID    -> 구독자     |
                |  ENDED  -> 구독자     |
                |  UNSOLD -> 구독자     |
                +-----------------------+
                              |
                              v
                           Client
```

| 패턴 | Auction | Bid | AuctionWatch | AuctionStatus |
|------|:-------:|:---:|:------------:|:-------------:|
| Rich Domain Model | O | - | - | - |
| Aggregate Root | O | - | - | - |
| Immutable Record | - | O | - | - |
| Toggle Pattern | - | - | O | - |

| ID 참조 (FK 없음) | O | O | O | - |
| DIP (Port/Adapter) | O | O | O | - |

| Event-Driven | O (Kafka 생성) | O (WS push) | - | - |

| Unique Constraint | - | - | O (auction+member) | - |
